## 阶段一：产品认识（0.5天）

### 1.1 快速理解 RAG 是什么

**一句话概括：** RAG = 检索增强生成。先从知识库中检索相关文档，再让 AI 基于这些文档生成回答。

**为什么需要 RAG：**
- LLM 的知识有截止日期，可能"胡说八道"
- 企业需要用私域知识（内部文档）来回答问题
- 需要答案有据可查，可追溯到原始文档

**阅读文档：**
- `docs/多跳技术方案.md`（10分钟，快速浏览）

### 1.2 理解系统能做什么

**核心功能（按用户价值排序）：**

| 功能 | 用户价值 |
|------|----------|
| 知识库问答 | 用户问问题，系统从文档中找答案 |
| 多格式文档支持 | 上传 PDF/Word/Excel，系统自动解析 |
| 多跳推理 | 复杂问题需要综合多个文档的信息 |
| 源文档溯源 | 回答标注文档来源，可信可查 |
| 评测系统 | 自动评估回答质量，持续优化 |
| 助手配置 | 自定义 AI 助手的角色和行为 |

## 阶段二：产品功能体验（0.5天）

### 2.1 系统部署与体验

**目标：** 亲手操作一遍系统，理解用户交互流程

**操作步骤：**
1. 启动所有服务（docker-compose up）
2. 注册账号、登录
3. 创建知识库
4. 上传一个 PDF 文档
5. 等待文档处理完成（查看状态）
6. 进入对话，提问相关问题
7. 查看回答的源文档
8. 查看评测功能

**需要理解的产品概念：**
- 知识库（Knowledge Base）：文档的集合
- 分块策略（Chunking）：文档如何被切分
- 会话（Session）：多轮对话的上下文
- 助手（Assistant）：可配置的 AI 角色

### 2.2 理解数据在各环节的形态

```
原始文档 (PDF/Word)
    ↓ 文档解析
文本内容 + 元数据（标题、页码）
    ↓ 文本分块
多个文本块 (Chunk)
    ↓ 向量化
向量 + 文本块 存储到 Milvus
    ↓ 用户提问
问题 → 检索 → 相关文本块
    ↓ 结合 Prompt
LLM → 生成回答
```

## 阶段三：核心概念深度理解（1天）

### 3.1 文档处理流程

**需要理解的概念：**

| 概念 | 产品含义 | 影响的指标 |
|------|----------|------------|
| **分块大小** | 每次检索多少文字 | 答案的完整性和相关性 |
| **分块策略** | 按什么规则切分 | 是否破坏语义完整性 |
| **Overlap** | 块与块之间的重叠 | 跨块信息能否被召回 |

**产品经理需要知道的：**
- 分块太小 → 答案可能片面、缺少上下文
- 分块太大 → 引入过多无关信息，LLM 容易"分心"
- 按语义分块 → 效果更好但实现复杂

### 3.2 检索与生成

**RAG 流程图：**
```
用户问题
    ↓
1. 检索：在知识库中找到最相关的文档块
    ↓
2. 重排序：精选最优质的候选块
    ↓
3. 生成：将检索结果作为上下文，让 LLM 生成回答
    ↓
回答 + 源文档引用
```

**你需要理解的三个关键点：**

1. **检索（Retrieval）**
   - 把问题转为向量，在向量数据库中找相似文本
   - 相似度越高 = 相关性越高
   - top_k = 返回多少个候选块

2. **重排序（Rerank）**
   - 初步检索结果可能不完美
   - 用更精准的模型重新排序
   - 选出最相关的几个块

3. **生成（Generation）**
   - 把检索到的文档作为"参考资料"
   - LLM 基于这些资料生成回答
   - 回答会标注来源文档

### 3.3 多跳推理（高级）

**什么是多跳：** 需要综合多个文档的信息才能回答的问题

**举例：**
- "A公司的2023年营收增长了多少？"（单跳）
- "A公司的2023年营收增长是因为什么？同期B公司有类似增长吗？"（多跳）

**产品价值：** 支持更复杂的问答场景

## 阶段四：效果评估体系（0.5天）

### 4.1 为什么需要评测

- RAG 效果好不好，用户说不清楚，需要量化指标
- 优化系统效果需要数据支撑
- 上线前需要验收标准

### 4.2 评测指标

| 指标 | 含义 | 产品视角 |
|------|------|----------|
| **Faithfulness** | 回答是否忠实于检索到的文档 | 答案有没有胡说八道 |
| **Answer Relevance** | 回答与问题的相关性 | 答案是否切题 |
| **Context Precision** | 检索到的文档是否精准 | 检索质量好不好 |
| **Context Recall** | 相关文档是否都被召回 | 有没有遗漏重要信息 |

### 4.3 评测产品设计

**QA 对生成：**
- 系统可以从文档自动生成问答对
- 也可以人工上传评测集
- 单跳 vs 多跳问题比例可配置

## 阶段五：系统架构与工程（0.5天）

### 5.1 为什么需要这些组件

| 组件 | 作用 | 如果没有会怎样 |
|------|------|----------------|
| **Milvus** | 存储和搜索向量 | 无法做语义检索 |
| **Redis** | 缓存对话历史 | 每次对话都是孤立的 |
| **MinIO** | 存储原始文档 | 文档无法管理 |
| **Celery** | 异步处理文档 | 大文档上传会超时 |
| **RabbitMQ** | 任务队列 | 无法并行处理 |

### 5.2 理解技术边界

**系统当前能力的上限：**
- 文档格式：PDF、Word、Excel、PPT、Markdown、TXT
- 单文档大小：受限于服务器内存
- 对话上下文：受限于 LLM 的上下文窗口
- 实时性：文档处理是异步的，有延迟

**系统的局限性：**
- 图片内容无法理解（除非用多模态模型）
- 复杂的表格结构解析效果有限
- 多跳推理效果不稳定
- 回答的准确性依赖文档质量

## 阶段六：竞品对比与产品思考（0.5天）

### 6.1 同类系统对比维度

| 维度 | 评估要点 |
|------|----------|
| 知识库管理 | 支持的格式、文档数量上限 |
| 检索效果 | 召回率、精确率、响应速度 |
| 生成质量 | 答案准确性、引用完整性 |
| 交互体验 | 上传流程、对话界面、报表 |
| 定制能力 | Prompt 调整、模型选择 |
| 部署方式 | SaaS / 私有化 / 混合部署 |
| 成本 | LLM API 费用、存储费用 |

### 6.2 产品规划思考点

**功能优化方向：**
- 更好的文档解析（表格、图表）
- 更好的多跳推理
- 更丰富的评测指标
- 更细粒度的知识库管理

**可能的商业模式：**
- 私有化部署（卖解决方案）
- SaaS 订阅（按知识库数量/用量收费）
- 定制化模型微调服务
