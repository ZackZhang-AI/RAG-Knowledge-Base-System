## 一、项目概述（0.5天）

### 1.1 系统定位
RAG（Retrieval-Augmented Generation）智库系统是一个**企业级知识库问答系统**，其核心价值在于：
- 将非结构化文档转化为可检索的知识
- 让 AI 回答基于真实的内部文档
- 提供答案的来源追溯能力

### 1.2 解决什么问题
| 痛点 | RAG 解决方案 |
|------|-------------|
| 企业知识分散在各种文档中 | 统一入库，集中管理 |
| AI 回答缺乏事实依据 | 基于文档检索生成 |
| 无法验证 AI 回答的准确性 | 提供引用来源追溯 |
| 文档更新后 AI 知识过时 | 重新向量化即可更新 |

### 1.3 核心使用场景
1. **内部知识库问答** - 员工查询公司制度、文档
2. **客服助手** - 基于产品文档回答客户问题
3. **技术文档问答** - 开发者查询 API 文档
4. **合同审查** - 自动提取合同关键条款

### 2.2 核心组件职责

| 组件 | 技术选型 | 职责 | 关键指标 |
|------|---------|------|---------|
| API 服务 | FastAPI | HTTP 请求处理 | QPS、响应延迟 |
| 任务队列 | RabbitMQ + Celery | 异步任务处理 | 任务吞吐量、失败率 |
| 向量数据库 | Milvus | 语义检索 | ANN 查询性能 |
| 关系数据库 | PostgreSQL (SQLAlchemy) | 业务数据存储 | 连接池、查询优化 |
| 缓存 | Redis | 会话缓存、热点数据 | 命中率、内存使用 |
| 对象存储 | MinIO | 文档文件存储 | 存储容量、IOPS |

### 2.3 数据流向

```
文档上传 → 文件存储(MinIO) → 异步解析 → 文本分块 → 向量化 → 存储(Milvus)
                              ↓
                        元数据存储(PostgreSQL)
```

```
用户提问 → 查询向量化 → 语义检索 → 重排序 → LLM 生成 → 返回答案
           ↓                    ↓
        Milvus 查询         引用来源标注
```

### 2.4 为什么这样设计

**问题1：为什么要用向量数据库？**
- 传统数据库只能做精确匹配（关键词搜索）
- 向量数据库可以做语义相似度搜索
- 示例：搜索"苹果"时，可以找到水果苹果和苹果手机

**问题2：为什么要用消息队列？**
- 文档处理是耗时操作（解析、向量计算）
- 同步处理会阻塞用户请求
- 异步处理可以削峰填谷

**问题3：为什么要用对象存储？**
- 文档文件很大（PDF、Word）
- 需要支持预览、下载、版本管理
- S3 协议是标准，MinIO 是私有部署方案

├── id: 主键

├── id: 主键

    │

        ├── id: 主键
        ├── session_id: 所属会话

        └── sources: 引用来源

├── name: 任务名称
├── dataset_type: 数据集类型
└── status: 任务状态
```

### 3.2 为什么这样设计

**KnowledgeBase → Document → Chunk 三层结构**
- 一个知识库可以包含多个文档
- 一个文档可以包含多个块（便于细粒度检索）
- Chunk 是向量化的最小单位

**ChatSession → ChatInteraction 设计**
- Session 是会话级别（一个对话窗口）
- Interaction 是每次问答（支持多轮对话）
- 支持对话历史分析和质检

## 四、文档处理流程（2天）

### 4.1 支持的文档格式

| 格式 | 解析方式 | 难度 |
|------|---------|------|
| PDF | PyMuPDF / pdfplumber | 中 |
| Word (.docx) | python-docx | 低 |
| Excel (.xlsx) | openpyxl | 中 |
| PPT (.pptx) | python-pptx | 中 |
| Markdown | 正则解析 | 低 |
| TXT | 直接读取 | 低 |

### 4.2 处理流程详解

```
步骤1: 文件上传
┌─────────┐
│ 用户上传 │ → 保存到 MinIO → 返回 file_path
└─────────┘

步骤2: 异步任务触发
┌────────────────────────┐
│ 发布 process_document  │
│ Task 到 RabbitMQ       │
└────────────────────────┘

步骤3: 文档解析 (Celery Worker)
┌────────────────────────────────────────┐
│ 1. 从 MinIO 下载文件                    │
│ 2. 根据格式选择解析器                   │
│ 3. 提取文本内容和元数据                 │
└────────────────────────────────────────┘

步骤4: 文本清洗
┌────────────────────────────────────────┐
│ - 去除乱码、空格规范化                  │
│ - 提取段落结构                          │
│ - 识别表格、标题                        │
└────────────────────────────────────────┘

步骤5: 文本分块 (Chunking)
┌────────────────────────────────────────┐
│ 策略: RecursiveCharacterTextSplitter  │
│ - 按段落/句子分割                       │
│ - 控制块大小（通常 500-1000 字）         │
│ - 块之间保留重叠（便于上下文连续性）     │
└────────────────────────────────────────┘

步骤6: 向量化
┌────────────────────────────────────────┐
│ 调用 DashScope API                      │
│ 输入: 文本块                            │
│ 输出: 1536 维度向量（text-embedding-v2）│
└────────────────────────────────────────┘

步骤7: 存储
┌────────────────────────────────────────┐
│ Milvus: 存储向量 + metadata            │
│ PostgreSQL: 存储块元数据                │
└────────────────────────────────────────┘
```

### 4.3 分块策略详解

**为什么需要分块？**
- LLM 有上下文长度限制（通常 4K-128K tokens）
- 完整文档太大，无法一次性输入
- 细粒度检索更精准

**分块方法对比**

| 方法 | 原理 | 适用场景 | 缺点 |
|------|------|---------|------|
| 固定大小 | 按字符数切分 | 通用 | 可能切断句子 |
| 段落分块 | 按段落分割 | 结构化文档 | 块大小不均匀 |
| 递归分块 | 按层级（段落→句子→词） | 通用 | 复杂度高 |
| 语义分块 | 按主题变化 | 长文档 | 实现复杂 |

**本项目采用: 递归字符分块 + 重叠**
- 优先按段落分割
- 段落太大则按句子分割
- 块之间保留 20% 重叠

## 五、向量化与检索（2天）

### 5.1 什么是文本向量化

```
文本: "今天天气很好"
    ↓ 向量化（Embedding）
向量: [0.123, -0.456, 0.789, ..., 0.234]  # 1536 维
```

**向量特性：**
- 语义相似的文本，向量也相似
- 可以通过向量距离衡量语义相似度
- 常用度量：余弦相似度、欧氏距离

### 5.2 向量数据库核心概念

**Milvus 核心对象：**

| 对象 | 说明 |
|------|------|
| Collection | 类似表，存储向量集合 |
| Partition | 分区，优化查询性能 |
| Field | 字段（向量 + 元数据） |
| Index | 索引，加速 ANN 查询 |

**向量检索原理：**
- ANN（Approximate Nearest Neighbor）近似最近邻
- 不做精确搜索，做速度和精度权衡
- 常用索引：HNSW、IVF

### 5.3 混合检索架构

```
用户查询
    │
    ▼
┌─────────────────┐
│ BM25 关键词检索  │ ← 传统全文检索，快速筛选
└────────┬────────┘
         │ top 100
         ▼
┌─────────────────┐
│ 向量相似度检索   │ ← 语义理解，精准匹配
└────────┬────────┘
         │ top 20
         ▼
┌─────────────────┐
│    重排序        │ ← Cross-Encoder 精排
└────────┬────────┘
         │ top 5
         ▼
    返回结果
```

**为什么需要混合检索？**
- 纯向量检索可能遗漏关键词匹配的内容
- 纯关键词检索无法理解语义
- 两者结合，取长补短

### 5.4 重排序（Rerank）

**初筛 vs 精筛：**
1. **向量检索（初筛）**：快速从百万级选出 top 100
2. **Rerank（精筛）**：用更精确的模型选出 top 5

**Cross-Encoder vs Bi-Encoder：**

| 方案 | 速度 | 精度 | 说明 |
|------|------|------|------|
| Bi-Encoder | 快 | 中 | 各自编码，cosine 相似度 |
| Cross-Encoder | 慢 | 高 | 拼接后联合编码 |

### 6.2 多跳查询流程

**什么是多跳问题？**
- 需要综合多个文档的信息才能回答
- 示例: "A公司的产品X和B公司产品Y对比，哪个更好？"
- 解决方案：拆解为多个子问题，分别检索，再综合

```
问题: "根据第三季度的财报和三季度的技术突破报告，分析公司未来的增长潜力"

    │
    ▼ 拆解
┌─────────────────────────────────┐
│ 子问题1: 第三季度财报的关键数据   │
│ 子问题2: 三季度的技术突破有哪些   │
│ 子问题3: 分析增长潜力             │
└─────────────────────────────────┘
    │
    ▼ 并行检索
┌─────────────────────────────────┐
│ 检索1: Q3财报 → 关键数据          │
│ 检索2: 技术报告 → 突破点          │
└─────────────────────────────────┘
    │
    ▼ 合并答案
┌─────────────────────────────────┐
│ 综合上下文 + 分析框架             │
│ → 生成最终回答                    │
└─────────────────────────────────┘
```

### 6.3 提示词工程（Prompt Engineering）

**RAG 场景的 Prompt 模板：**

```
你是专业的知识库问答助手。请根据以下上下文信息回答用户问题。

## 上下文信息
{retrieved_chunks}

## 用户问题
{user_query}

## 回答要求
1. 仅基于提供的上下文信息回答，不要编造信息
2. 如上下文中没有相关信息，请明确说明"我无法从知识库中找到答案"
3. 在回答中引用相关文档来源，格式：[文档名-页码]
4. 回答要准确、简洁、有条理
5. 如果上下文信息不足，可以结合你的知识补充，但要说明

## 回答格式
回答内容
---
来源：
- [来源1]
- [来源2]
```

**调优参数：**

| 参数 | 说明 | 建议值 |
|------|------|-------|
| Temperature | 创造性/准确性 | 0.3（准确）~ 0.7（创造） |
| Top-P | 采样范围 | 0.9 |
| Max Tokens | 最大生成长度 | 根据问题复杂度 |

## 七、评测系统（1天）

### 7.1 为什么要评测

RAG 系统的效果评估至关重要，因为：
- 检索不好 → 回答无依据
- 生成不好 → 回答不准确
- 端到端评估才能发现真实问题

### 7.2 RAGAS 评测框架

**核心指标：**

| 指标 | 衡量内容 | 计算方式 |
|------|---------|---------|
| Faithfulness | 回答是否忠于检索内容 | 生成答案中的事实有多少被上下文支持 |
| Answer Relevance | 回答是否和问题相关 | 回答内容与问题的语义相似度 |
| Context Precision | 上下文排序是否正确 | 相关块是否排在前面 |
| Context Recall | 相关上下文是否都被召回 | 正确答案中的事实是否都在上下文中 |

### 7.3 评测流程

```
评测任务创建
    │
    ▼
┌─────────────────────────────────────┐
│ 生成评测数据集（QA对自动生成）        │
│ - 单跳问题：直接从文档生成            │
│ - 多跳问题：综合多个文档生成           │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 运行 RAG 流程                        │
│ - 对每个问题执行检索+生成             │
│ - 记录检索结果和生成答案              │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 计算评测指标                          │
│ - 调用 LLM 评估每项指标               │
│ - 汇总生成评测报告                    │
└─────────────────────────────────────┘
```

└── 评测中心 (/evaluation)
    ├── 评测任务列表
    ├── 创建评测任务
    │   ├── 数据集选择
    │   ├── 评测范围
    │   └── 启动评测
    └── 评测报告查看
```

### 8.2 核心交互流程

**文档上传流程：**
```
用户点击上传 → 选择文件 → 显示上传进度
    ↓
前端调用 POST /api/documents/upload
    ↓
后端返回 task_id
    ↓
前端轮询 GET /api/documents/{id}/status
    ↓
状态变为 "completed" → 显示文档列表
```

**对话流程：**
```
用户输入问题 → 点击发送
    ↓
前端 POST /api/chat/stream（WebSocket）
    ↓
后端流式返回 chunks
    ↓
前端实时显示回答
    ↓
回答完成 → 显示引用来源
```
